## Context

当前 `src/services/aiService.ts` 中的 `buildPrompt` 函数生成一段纯文本提示词，以单条 user message 发送给 AI。提示词风格偏向高级模型——使用抽象指令（"写一份极其专业、充满启发性的报告"）、无格式约束、无示例引导。

项目已支持 GLM-4-Flash 免费方案，用户可能使用 GPT-3.5-Turbo、Qwen-Turbo、GLM-4-Flash 等能力较弱的模型。这些模型在面对开放式指令时常见问题：
- 输出格式不稳定（有时用列表、有时用段落、有时混用）
- 章节遗漏或合并
- 内容空泛、缺乏针对性
- 语气不一致

## Goals / Non-Goals

**Goals:**
- 让 GLM-4-Flash / GPT-3.5 级别模型也能输出结构完整、格式一致、内容有针对性的报告
- 保持高级模型（GPT-4o、Claude 3.5）的报告质量不退化
- 提示词方案对中英文双语均有效

**Non-Goals:**
- 不做模型路由/自动降级（不根据模型名自动切换提示词版本）
- 不修改 UI 展示逻��
- 不增加新的 API 调用（如多轮对话、chain-of-thought 拆分）
- 不引入 prompt template 库或新依赖

## Decisions

### Decision 1: 使用 system + user 角色分离

**选择**: 将当前单条 user message 拆分为 system message（角色设定 + 输出规则）和 user message（具体任务 + 用户数据）。

**理由**: 低级模型对 system message 中的指令遵从度显著高于 user message 中的混合指令。角色分离让模型更容易区分"我是谁"和"我要做什么"。

**替代方案**: 保持单条 message 但加强格式约束——对 GLM-4-Flash 测试效果不佳，格式遵从率低。

### Decision 2: 提供 Markdown 输出骨架模板

**选择**: 在提示词中给出完整的 Markdown 输出结构骨架，包含每个章节的标题和占位说明。

**理由**: 低级模型擅长"填空"而非"创作"。给定骨架后，模型只需在每个位置填充内容，大幅降低结构混乱的概率。

**替代方案**: 仅用编号列表描述章节——当前方案即是如此，低级模型经常遗漏章节或格式不一致。

### Decision 3: 添加字数范围约束和负面约束

**选择**: 为每个章节指定大致字数范围（如"150-300字"），并添加明确的负面约束（"不要使用空泛的鼓励语"、"不要重复优势名称的定义"）。

**理由**: 低级模型容易产出过短（一句话带过）或过长（大段废话）的内容。字数约束提供了锚点。负面约束减少常见的低质量模式。

**替代方案**: 仅用 few-shot 示例引导——token 消耗过大，且低级模型的 context window 有限。

### Decision 4: 不区分模型层级，统一使用优化后的提示词

**选择**: 所有模型使用同一套优化后的提示词，不做条件分支。

**理由**: 
- 结构化提示词对高级模型无负面影响（高级模型能很好地遵循结构化指令）
- 减少维护成本，避免模型名判断逻辑的脆弱性
- 用户可能使用自定义模型名，无法可靠判断能力层级

**替代方案**: 根据模型名维护两套提示词——增加复杂度，且模型名不可靠。

### Decision 5: 调整 temperature 为 0.6

**选择**: 将 temperature 从 0.7 降至 0.6。

**理由**: 低级模型在较高 temperature 下更容易跑题和格式混乱。0.6 在保持一定创造性的同时提升输出稳定性。对高级模型影响可忽略。

## Risks / Trade-offs

- **提示词 token 增加** → 优化后的提示词约 800-1000 tokens（当前约 200-300 tokens），增加约 500-700 tokens 的输入成本。对于免费方案用户（GLM-4-Flash）这是可接受的。→ 缓解：保持骨架精简，不使用 few-shot 完整示例。
- **高级模型报告风格可能略有变化** → 结构化骨架可能让高级模型的输出更"模板化"。→ 缓解：骨架中使用"参考结构"而非"严格遵循"的措辞，给高级模型留有发挥空间。
- **中英文提示词维护成本翻倍** → 两套语言的提示词都需要同步更新。→ 缓解：保持两套提示词结构一致，仅翻译内容。
